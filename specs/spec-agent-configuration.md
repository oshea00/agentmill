# Spec: Agent Configuration & Change Management

## Overview

Agent configuration is defined in Markdown files with YAML frontmatter. There
are two kinds of configuration file: **templates** (human-authored, committed to
git, version-controlled) and **instances** (orchestrator-generated at runtime,
derived from a template). Git is the single source of truth for templates.
Instances are ephemeral by default and not committed.

---

## File Format

Every agent config file is a `.md` file with a YAML frontmatter block followed
by a freeform Markdown body. The frontmatter holds all structured, machine-read
fields. The Markdown body is the agent's **system prompt** — it is passed
verbatim to Claude as the system message for that agent's session.

```
agents/
  templates/
    researcher.md
    coder.md
    summariser.md
  instances/               ← generated at runtime, gitignored by default
    <agent-id>.md
```

### Full template example: `agents/templates/researcher.md`

```markdown
---
# Required fields
name: researcher
version: "1.2.0"
role: worker                     # orchestrator | worker
execution_mode: messages_api     # messages_api | claude_code

# Tool access
tools:
  native:
    - web_search
    - read_file
  mcp:
    - brave-search
    - filesystem

# Execution limits
limits:
  max_iterations: 20
  max_tool_calls: 50
  tool_timeout_seconds: 30
  session_timeout_seconds: 300

# Peer relationships (worker only)
peers:
  sync_enabled: true
  sync_type: partial_result      # partial_result | context_share

# Allowed filesystem paths (applied to native tools + claude_code sessions)
allowed_paths:
  - ./workspace
  - ./data

# Environment variables this agent requires (validated at instantiation)
required_env:
  - ANTHROPIC_API_KEY
  - BRAVE_API_KEY

# Optional metadata
tags:
  - research
  - web
description: "Researches topics using web search and local files."
---

## Role

You are a research agent. Your job is to gather accurate, well-sourced
information on the topic you are assigned and return a structured summary.

## Behaviour

- Always cite your sources with URLs.
- If a search returns no useful results, try at least two alternative queries
  before reporting failure.
- Prefer primary sources (official docs, papers, government sites) over
  aggregators.
- Do not fabricate information. If you cannot find an answer, say so.

## Output Format

Return your findings as a Markdown document with the following sections:
- **Summary** (2–3 sentences)
- **Key Findings** (bullet list)
- **Sources** (URL list)
```

## YAML Frontmatter Schema

### Required fields

| Field | Type | Values | Description |
|---|---|---|---|
| `name` | string | identifier | Canonical name; must be unique within `templates/` |
| `version` | string | semver `"MAJOR.MINOR.PATCH"` | Template version |
| `role` | enum | `orchestrator` \| `worker` | Agent role in the topology |
| `execution_mode` | enum | `messages_api` \| `claude_code` | How Claude is invoked |

### Optional fields

| Field | Type | Default | Description |
|---|---|---|---|
| `tools.native` | list[string] | `[]` | Native tool names from `tools/builtin.py` |
| `tools.mcp` | list[string] | `[]` | MCP server names from `mcp_servers.json` |
| `limits.max_iterations` | int | `20` | Max Claude turns per task |
| `limits.max_tool_calls` | int | `50` | Max tool calls per task |
| `limits.tool_timeout_seconds` | float | `30.0` | Per-tool call timeout |
| `limits.session_timeout_seconds` | float | `300.0` | Total task timeout |
| `peers.sync_enabled` | bool | `false` | Whether this agent participates in peer sync |
| `peers.sync_type` | enum | `partial_result` | Type of peer sync messages to send |
| `allowed_paths` | list[string] | `[]` | Filesystem paths available to this agent |
| `required_env` | list[string] | `[]` | Env vars that must be set at instantiation |
| `tags` | list[string] | `[]` | Free-form labels for filtering/search |
| `description` | string | `""` | One-line summary (shown in logs and UIs) |

## Templates vs Instances

### Templates

- Live in `agents/templates/`.
- Authored by humans; committed to git.
- Define the *type* of agent — its capabilities, limits, and system prompt.
- A template is a blueprint, not a running agent.
- Templates are **immutable once committed** — changes require a version bump
  (see Change Management below).

### Instances

- Live in `agents/instances/<agent-id>.md` at runtime.
- Generated by the orchestrator when it spawns a worker.
- Inherit all fields from the template, then have **task-specific fields
  injected** into the frontmatter:

```yaml
# Injected by orchestrator at spawn time — not present in templates
_instance:
  agent_id: "3f2a1c..."           # UUID v4
  template: researcher            # source template name
  template_version: "1.2.0"      # version at spawn time (snapshot)
  parent_id: "a1b2c3..."         # orchestrator's agent ID
  cluster_id: "cluster-xyz"
  task_id: "task-001"
  task_description: "Research Q3 earnings for ACME Corp"
  spawned_at: "2026-03-28T14:03:01Z"
```

The instance Markdown body is the template body with any `{{ variable }}`
placeholders resolved (see Templating below). The resolved body becomes the
system prompt for that agent's Claude session.

## `AgentConfigLoader`

Responsible for reading, validating, and converting config files into the
runtime `Agent` dataclass and `context` dict.

```python
from dataclasses import dataclass
from typing import Optional
import yaml, re
from pathlib import Path

FRONTMATTER_RE = re.compile(r"^---\n(.*?)\n---\n(.*)", re.DOTALL)

@dataclass
class AgentConfig:
    # Parsed from frontmatter
    name: str
    version: str
    role: str
    execution_mode: str
    tools_native: list[str]
    tools_mcp: list[str]
    limits: dict
    peers: dict
    allowed_paths: list[str]
    required_env: list[str]
    tags: list[str]
    description: str
    # Parsed from body
    system_prompt: str
    # Instance-only (None for templates)
    instance_meta: Optional[dict] = None

class AgentConfigLoader:
    def load_template(self, name: str) -> AgentConfig:
        """Load and validate a template by name."""
        path = Path(f"agents/templates/{name}.md")
        return self._parse(path)

    def load_instance(self, agent_id: str) -> AgentConfig:
        """Load an instance config by agent ID."""
        path = Path(f"agents/instances/{agent_id}.md")
        return self._parse(path)

    def _parse(self, path: Path) -> AgentConfig:
        raw = path.read_text()
        match = FRONTMATTER_RE.match(raw)
        if not match:
            raise ConfigParseError(f"No YAML frontmatter found in {path}")
        front = yaml.safe_load(match.group(1))
        body  = match.group(2).strip()
        self._validate(front, path)
        return self._to_config(front, body)

    def _validate(self, front: dict, path: Path) -> None:
        """Raise ConfigValidationError on missing required fields or bad values."""
        required = ["name", "version", "role", "execution_mode"]
        for field in required:
            if field not in front:
                raise ConfigValidationError(f"Missing required field '{field}' in {path}")
        if front["role"] not in ("orchestrator", "worker"):
            raise ConfigValidationError(f"Invalid role '{front['role']}' in {path}")
        if front["execution_mode"] not in ("messages_api", "claude_code"):
            raise ConfigValidationError(f"Invalid execution_mode in {path}")
        # Validate semver
        parts = str(front["version"]).split(".")
        if len(parts) != 3 or not all(p.isdigit() for p in parts):
            raise ConfigValidationError(f"version must be semver (MAJOR.MINOR.PATCH) in {path}")

    def _to_config(self, front: dict, body: str) -> AgentConfig:
        tools   = front.get("tools", {})
        limits  = {**{"max_iterations": 20, "max_tool_calls": 50,
                       "tool_timeout_seconds": 30.0, "session_timeout_seconds": 300.0},
                   **front.get("limits", {})}
        peers   = {**{"sync_enabled": False, "sync_type": "partial_result"},
                   **front.get("peers", {})}
        return AgentConfig(
            name=front["name"], version=front["version"],
            role=front["role"], execution_mode=front["execution_mode"],
            tools_native=tools.get("native", []),
            tools_mcp=tools.get("mcp", []),
            limits=limits, peers=peers,
            allowed_paths=front.get("allowed_paths", []),
            required_env=front.get("required_env", []),
            tags=front.get("tags", []),
            description=front.get("description", ""),
            system_prompt=body,
            instance_meta=front.get("_instance"),
        )
```

## Templating (System Prompt Variables)

The Markdown body may contain `{{ variable }}` placeholders. The orchestrator
resolves these at spawn time before writing the instance file.

```markdown
## Task

Your assigned task is: {{ task_description }}

You are working in cluster {{ cluster_id }} alongside {{ peer_count }} peer agents.
```

Resolved by the orchestrator:

```python
def resolve_system_prompt(template_body: str, variables: dict) -> str:
    for key, value in variables.items():
        template_body = template_body.replace("{{ " + key + " }}", str(value))
    # Fail loudly on any unresolved placeholder
    unresolved = re.findall(r"\{\{.*?\}\}", template_body)
    if unresolved:
        raise ConfigTemplatingError(f"Unresolved placeholders: {unresolved}")
    return template_body
```

Standard variables available at spawn time:

| Variable | Source |
|---|---|
| `{{ task_description }}` | `TASK_ASSIGN` message payload |
| `{{ task_id }}` | `TASK_ASSIGN` message payload |
| `{{ agent_id }}` | Generated UUID |
| `{{ cluster_id }}` | Orchestrator's cluster |
| `{{ peer_count }}` | Number of co-workers in cluster |
| `{{ parent_name }}` | Orchestrator's template name |

## Orchestrator Spawn Flow

```
Orchestrator receives task requiring a worker
    │
    ├─ AgentConfigLoader.load_template(name)
    ├─ Validate required_env are set in os.environ
    ├─ resolve_system_prompt(body, variables)
    ├─ Write instance file → agents/instances/<agent-id>.md
    ├─ AgentRepository.create(agent)            ← lifecycle spec
    ├─ AgentRepository.transition(INITIALIZING) ← lifecycle spec
    └─ MessageBus.send(TASK_ASSIGN)             ← messaging spec
```

## Change Management

### Git is the source of truth

All template files are committed to git. The authoritative state of any agent
type is whatever is on the main branch at the time of deployment.

### Versioning rules

Template versions follow **semver**. Change type determines version increment:

| Change type | Bump | Example |
|---|---|---|
| New optional field, wording tweaks in system prompt | PATCH | `1.2.0 → 1.2.1` |
| New required field, new tool added or removed, limit changes | MINOR | `1.2.0 → 1.3.0` |
| Role change, execution_mode change, breaking schema change | MAJOR | `1.2.0 → 2.0.0` |

### Compatibility policy

- A running instance is **pinned to the template version it was spawned from**
  (stored in `_instance.template_version`).
- Template changes do **not** affect running instances — only newly spawned
  agents pick up the new version.
- The orchestrator logs a warning if it spawns an agent from a template whose
  version differs from the version used by peer agents in the same cluster.

### Reviewing changes

Before merging a template change, check:

```bash
git diff agents/templates/researcher.md
```

If the system prompt changed materially, run the agent against a regression
task set before promoting to main. There is no automated prompt regression
testing in v1 — this is a manual step.

### Deprecating a template

1. Add `deprecated: true` to the frontmatter.
2. Commit and merge.
3. The orchestrator logs a `WARNING` when loading a deprecated template but
   does not refuse to spawn from it.
4. After all production clusters have been drained of the deprecated type,
   delete the file and bump the framework's minor version.

## Validation at Startup

On framework startup (before any agent is spawned), `AgentConfigLoader` scans
all files in `agents/templates/` and validates each one. Any validation failure
is a hard startup error — the process exits rather than running with a broken
template.

```python
def validate_all_templates(templates_dir: str = "agents/templates") -> None:
    loader = AgentConfigLoader()
    errors = []
    for path in Path(templates_dir).glob("*.md"):
        try:
            loader.load_template(path.stem)
        except (ConfigParseError, ConfigValidationError) as e:
            errors.append(str(e))
    if errors:
        raise StartupValidationError("\n".join(errors))
```

particular run for debugging purposes.

## Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Format | Markdown + YAML frontmatter | Frontmatter is machine-readable; body doubles as system prompt documentation |
| Template vs instance | Separate files, separate dirs | Clear boundary between human-authored config and runtime artefacts |
| System prompt location | Markdown body | Keeps prompt editable as prose, not a JSON string |
| Versioning | Semver on template files | Git blame + version number gives full change history without a separate changelog |
| Instance mutability | Immutable after spawn | Prevents mid-run config drift; re-spawn for new config |
| Startup validation | Hard fail on invalid template | Fail fast; never run with a silently broken agent definition |
| Placeholder syntax | `{{ variable }}` | Readable in raw Markdown; easy to grep for unresolved vars |
